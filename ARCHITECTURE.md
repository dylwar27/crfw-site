# ARCHITECTURE.md — how the CRFW site fits together

The CRFW site is a static Astro 4 build that projects a file-based archive into a timeline. This doc covers the moving parts: content collections, the vault pipeline, the bucket timeline, the filter system, the SQLite projection, and the build chain. For volatile facts (counts, script inventory) see `CLAUDE.md` STATUS block or run `node scripts/status.mjs`.

---

## 1. The three layers

```
┌───────────────────────────────────┐
│ Dropbox archive (the truth)       │ CRFW Archive/… — audio, video, images,
│ ~125 GB / ~45,000 files           │ scans, Whisper sidecars, Word docs, IG
│                                   │ scrapes, xlsx snapshots, vault notes.
└──────────────┬────────────────────┘
               │ archivePath: "CRFW Archive/…" on every entry
               ▼
┌───────────────────────────────────┐
│ src/content/ (the repo copy)      │ .md / .json, validated at build time
│ site collections + projected vault│ against Zod schemas in config.ts.
└──────────────┬────────────────────┘
               │ astro build → HTML + Pagefind + SQLite
               ▼
┌───────────────────────────────────┐
│ dist/ (the presentation layer)    │ Static HTML, CSS, JS, Pagefind index,
│ shipped to GitHub Pages           │ SQLite projection at data/crfw.db.
└───────────────────────────────────┘
```

The Dropbox archive is the truth. The repo is a curator-editable presentation layer. The database is a read-only projection. If the DB dies nothing is lost; if the repo dies the archive and vault are still intact.

## 2. Content collections

Schemas live in [`src/content/config.ts`](src/content/config.ts). All collections share a `published` boolean that defaults to `true` — except `photos`, which defaults to `false` (draft until explicitly approved).

### Site collections (curator-editable)
| collection | file | purpose |
|---|---|---|
| `releases` | `.md` | LPs, EPs, singles, mixes, demos, b-sides, compilations |
| `photos` | `.json` | still images — IG scrapes + archive scans |
| `videos` | `.json` | music videos, DIMCP chunks, YouTube drafts, IG videos |
| `voice_memos` | `.json` | Whisper-transcribed iPhone memos Sept 2015 – Dec 2016 |
| `events` | `.md` | life events + press mentions |
| `sets` | `.json` | asset sets (PR #34) — grouped photo/video bundles |

### Vault projections (read-only — regenerated by `sync-vault.mjs`)
| collection | source in `_Vault/` |
|---|---|
| `vault_people` | `people/*.md` |
| `vault_projects` | `projects/*.md` |
| `vault_press` | `press/*.md` |
| `vault_venues` | `venues/*.md` |
| `vault_organizations` | `organizations/*.md` |
| `vault_funds` | `funds/*.md` |
| `vault_grants` | `grants/*.md` |
| `vault_releases` | `releases/*.md` — reconciled with site `releases` |
| `vault_tracks` | `tracks/*.md` |
| `vault_events` | `events/*.md` |
| `vault_series` | `series/*.md` |

Vault projections must never be hand-edited in `src/content/vault_*/` — edit the vault source and re-run `sync-vault.mjs`. See `VAULT.md` for authority rules.

## 3. The bucket timeline (Session 18)

`src/pages/index.astro` renders 8 bucket kinds to compress ~1,960 entries into something scannable. Buckets are synthesized in the page template from the raw entries list.

| kind | what it groups | trigger |
|---|---|---|
| `release_block` | everything attached to one release (cover + tracks + related media) | release entity |
| `dimcp_chunk` | DIMCP video series grouped by chunk number | video `project == "DIMCP"` |
| `youtube_strip` | standalone YouTube videos with CDN thumbnails | video with `youtubeId` |
| `photo_carousel` | up to N photos per year as a preview strip | photos year bucket |
| `voice_memo_group` | "Month YYYY · N voice memos" card | voice memos by year+month |
| `press_strip` | press events bundled by year | events with `kind == "press"` |
| `unreleased_group` | unreleased tracks/demos pooled | heuristics |
| `misc` | fallback for anything else | — |

Popup payload is a JSON blob embedded in the page (line ~665 of `src/pages/index.astro`) that the popup script uses to hydrate detailed views on click. See the Session 18 commit `1881c68` for full bucket synthesis code.

## 4. The filter system

Four dropdown axes with live counts and URL-param pre-selection (`?project=X&tag=Y`). Session 11 shipped.

| axis | values |
|---|---|
| Project | alphabets, killd by, Colin Ward, collaboration, DIMCP, life, other |
| Format | LP, EP, single, mix, demo, b-sides, other, compilation |
| Medium | release, event, photo, video, voice_memo, lyrics |
| Tag | auto-scaled — recomputes scoped to other-axis selections |

The tag axis is special: it recomputes whenever the other three change, so the tag list always reflects "tags present in the currently filtered subset." A `/tags` page groups all tags by source (curator / script / hashtag).

Filter state is encoded in URL params so links are shareable.

## 5. The vault pipeline

Five scripts, chained via `npm run build`:

```
 sync-vault.mjs --write
        │ projects _Vault/*.md → src/content/vault_*/*.json
        ▼
 reconcile-vault-releases.mjs
        │ merges vault_releases into site releases; BIN files for displaced text
        ▼
 astro build
        │ renders 686 pages
        ▼
 pagefind --site dist …
        │ indexes HTML to dist/pagefind/
        ▼
 db-sync.mjs --dist --quiet
        │ projects into data/crfw.db and copies to dist/data/crfw.db
```

Each script is idempotent with respect to current vault/content state. See `SCRIPT_PIPELINE.md` for per-script signatures and dry-run defaults.

## 6. Entity pages

| route | source | count |
|---|---|---|
| `/person/[slug]` | `vault_people` | one per entry |
| `/people` | index | 1 |
| `/press/[slug]` | `vault_press` | one per entry |
| `/press` | index | 1 |
| `/project/[slug]` | site `releases` project bucketing | one per project |
| `/voice-memo/[slug]` | `voice_memos` | 305 |
| `/video/[slug]` | `videos` with transcript | ~266 |
| `/about`, `/admin`, `/tags`, `/projects` | static pages | 4 |

Not yet paged: `vault_venues`, `vault_organizations`, `vault_releases` (standalone), `vault_tracks`, `vault_events`, `vault_funds`, `vault_grants`, `vault_series`. Those render as `.wiki-chip` badges when wikilinked.

Wikilink routing is defined in `src/lib/renderBody.ts`:
```
people → /person      projects → /project      press → /press
venues → /venue*      organizations → /org*    funds → /fund*
grants → /grant*      tracks/releases/events/series → chip only
  (* = collection exists in WIKILINK_ROUTES but page generator is not implemented;
       wikilinks currently render as chip)
```

See `CONVENTIONS.md#wikilinks` for wikilink syntax.

## 7. The SQLite projection

`scripts/db-sync.mjs` reads `src/content/**` and writes `data/crfw.db` (copied to `dist/data/crfw.db` for serving). Schema in `data/schema.sql`. FTS5 indexed.

Tables roughly mirror collections (`releases`, `photos`, `videos`, `voice_memos`, `events`, `sets`, plus `vault_*` variants). Cross-refs are join rows — zero dead refs on the last full build.

Used by `/admin` and for ad-hoc curator queries. Not required at runtime — the site works without it.

---

## 8. Build output

`npm run build` produces `dist/`:
- HTML pages (one per timeline route, entity page, reader page)
- `/pagefind/` client-side full-text search index
- `/data/crfw.db` SQLite projection
- `/media/…` copied static assets

Deploy target: GitHub Pages via `.github/workflows/deploy.yml` using `withastro/action@v3`. `base: '/crfw-site'` in `astro.config.mjs` means all internal links must route through the `withBase()` helper. A future migration to Cloudflare Pages (see plan's Step 4) would drop this.

## 9. Other important code

- `src/lib/renderBody.ts` — markdown + wikilink renderer with stash/restore tokenizer. Used on every entity body.

- `scripts/curators-kit/` — the internal CMS (`npm run cms`) that edits both site content and the vault. See `CURATOR_GUIDE.md`.
