# VAULT.md — the Obsidian vault, its relationship to the site, and the authority rules

The CRFW archive has two curator surfaces: **site content** (`src/content/*`) and **the vault** (`_Vault/` inside the Dropbox archive). This doc explains how they interact, who wins on conflict, and what the agent may or may not touch.

---

## 1. What the vault is

An Obsidian-style vault at:
```
~/Library/CloudStorage/Dropbox/CRFW/CRFW Archive/_Vault/
```

It holds structured entity notes as one Markdown file per entity, organized by kind:
```
_Vault/
├── people/
├── projects/
├── press/
├── venues/
├── organizations/
├── funds/
├── grants/
├── releases/
├── tracks/
├── events/
└── series/
```

Each note has YAML frontmatter + Markdown body. The body uses Obsidian wikilinks (`[[people/colin-ward]]`). See `CONVENTIONS.md#wikilinks`.

Extras in the vault root (not projected into the site):
```
_Vault/README.md        — curator overview (see below)
_Vault/SCHEMA.md        — entity schemas
_Vault/INDEX.md         — entity index
_Vault/DAILY_LOG.md     — curator's daily notes
_Vault/RESEARCH_SOURCES.md
_Vault/VALIDATION_REPORT.md
```

The vault is **Dropbox-synced, not git-tracked**. That's intentional — it's the curator's live working notebook. The site repo gets a *projection* of it.

---

## 2. The authority rule (one sentence)

**The vault wins on conflict; displaced site-side curator text goes to a `.BIN.md` file.**

Expand:
- The vault is the source of truth for everything that has a vault entry (people, projects, press, venues, orgs, funds, grants, releases, tracks, events, series).
- The site repo's `src/content/vault_*/` is a read-only projection of the vault — regenerated by `scripts/sync-vault.mjs`.
- Where vault and site overlap (releases — both exist), the reconciler (`scripts/reconcile-vault-releases.mjs`) merges vault fields into site release stubs.
- If the site release had hand-authored `summary` / `tags` text that the vault is now replacing, the old text is preserved in `<slug>.BIN.md` next to the release.

Nothing is ever silently lost. But there is no undo tool for BIN files yet — see §6.

## 3. What the agent may / may not touch

### MAY (with normal judgment)
- Edit site content in `src/content/releases/`, `photos/`, `videos/`, `voice_memos/`, `events/`, `sets/`.
- Run `sync-vault.mjs` and `reconcile-vault-releases.mjs`.
- Commit the projections that those scripts write into `src/content/vault_*/`.
- Read vault files for context (grep, cross-reference).
- Write new site stubs when reconciliation says "unmatched" — see `SLUG_MAP_TRIAGE.md`.

### MAY NOT (without explicit user direction)
- Edit files inside `_Vault/` directly. The vault is Dyl's curator surface. Exception: the agent may update `_Vault/README.md` when the user asks for doc work.
- Hand-edit files in `src/content/vault_*/`. Those are projections — edits are overwritten on the next sync. Edit the vault source instead, then re-sync.
- Delete or rename vault entries (golden rule #1).
- Run `--write` on pipeline scripts without understanding the dry-run first.

### MUST
- Warn when `reconcile-vault-releases.mjs` emits unmatched slugs — even dry-run. Triage per `SLUG_MAP_TRIAGE.md`.
- Preserve `preservedTitle` and `archivePath` on any merged entity.
- Use `CONVENTIONS.md#wikilinks` syntax when writing wikilinks into new content.

## 4. Projection (`scripts/sync-vault.mjs`)

Reads `_Vault/**/*.md`, projects each entity into `src/content/vault_<kind>/<slug>.json`.

**Dry-run by default.** Pass `--write` to apply. Example:
```bash
node scripts/sync-vault.mjs                 # dry-run — reports only
node scripts/sync-vault.mjs --write         # apply
```

Idempotent with respect to vault state — running twice against the same vault yields the same output.

Output is deterministic JSON; ordering is stable so git diffs are clean.

Also prints:
- Total entities projected, per kind.
- Wikilinks found and unique link targets.
- **Dead wikilinks** — refs to slugs that don't exist. Investigate; they point at vault authoring bugs.

---

## 5. Reconciliation (`scripts/reconcile-vault-releases.mjs`)

Merges `vault_releases/*.json` into site `releases/*.md`. Fields merged: `date`, `format`, `bandcampUrl`, `tracklist` (with preservedTitle), `summary`, `tags`, `archivePath`.

**Dry-run by default.** Pass `--write` to apply. Example:
```bash
node scripts/reconcile-vault-releases.mjs                    # dry-run — reports only
node scripts/reconcile-vault-releases.mjs --write            # apply
node scripts/reconcile-vault-releases.mjs --vault-slug X     # single entry
```

Chained after every `vault:sync` and `build` in dry-run mode (Session 17) — warnings print to stderr but don't block the build.

**Conflict handling:**
- Vault value wins.
- If the site stub had curator-authored prose in a field the vault is now replacing, the displaced content is written to `src/content/releases/<slug>.BIN.md`.
- BIN files look like:
  ```
  ---
  displaced_from: releases/<slug>.md
  displaced_at: 2026-04-23T12:34:56Z
  fields: [summary, tags]
  ---

  [original summary text]

  ORIGINAL TAGS: killd-by, b-sides
  ```
- BIN files are git-tracked in `reconcile-bin/`.

**Unmatched warning:**
If a vault release has no corresponding site release stub, the reconciler emits a WARNING listing slugs for SLUG_MAP triage. See `SLUG_MAP_TRIAGE.md`.

## 6. BIN recovery (manual — no script yet)

There is no `reconcile-vault-releases.mjs --rollback` yet. To restore a displaced value by hand:

1. Find the BIN file: `ls src/content/releases/*.BIN.md` (or wherever they land — vault release BINs go to `reconcile-bin/`).
2. Open it, read the displaced content.
3. Decide: keep the curator version, keep the vault version, or merge.
4. If restoring the curator version: copy the content back into the site `.md` frontmatter, commit, and update the vault entry so the next sync doesn't re-displace it.

Building a rollback tool is a candidate for a future session — flagged in the plan's "deferred" list.

## 7. Pages vs. chips

The site renders wikilinks to different vault collections differently:

| collection | page route | renders in body text as |
|---|---|---|
| `people` | `/person/[slug]` | link with avatar chip |
| `projects` | `/project/[slug]` | link with project badge |
| `press` | `/press/[slug]` | link |
| `venues` | — (no page) | `.wiki-chip` |
| `organizations` | — (no page) | `.wiki-chip` |
| `funds` | — (no page) | `.wiki-chip` |
| `grants` | — (no page) | `.wiki-chip` |
| `releases` | site `/release/[slug]` in popup | `.wiki-chip` |
| `tracks` | — (no page) | `.wiki-chip` |
| `events` | — (no page) | `.wiki-chip` |
| `series` | — (no page) | `.wiki-chip` |

Pages for venues/organizations/funds/grants/tracks/events/series are candidates for future sessions. The `.wiki-chip` render is an interim until pages exist.

## 8. Vault authoring conventions (for the curator)

When Dyl writes a new vault entry in Obsidian, he should follow:
- Frontmatter at the top with `id`, `kind`, `title`, required kind-specific fields (see `_Vault/SCHEMA.md`).
- Body in Markdown below `---`.
- Wikilinks use `[[coll/slug]]` syntax from `CONVENTIONS.md#wikilinks`.
- `archivePath` if the entity has a source folder.
- `preservedTitle` if the entity name has Colin's typography.
- `created` / `updated` ISO dates.

After curator edits land in Obsidian (synced via Dropbox), the next `node scripts/sync-vault.mjs --write` projects them into the site. The curator does not need to run the script — it runs automatically on `npm run build`.

See `_Vault/README.md` (inside Dropbox) for the curator-facing version of these conventions.

---

## 9. Current vault state

Run `node scripts/status.mjs` for a fresh count of vault source ↔ projection. Parity between columns is the healthy signal — drift means sync-vault hasn't run lately.

Dead wikilinks in sync-vault output are authoring bugs in the vault (typo, missing target entity). They don't block the build but should be cleaned up as part of curator work.
